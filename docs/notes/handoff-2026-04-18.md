## What's not done

### `Modifier` wiring — the next big lift

The math kernels are all ported and unit-tested, but they take normalized `(x, y)` floats. The upstream `lfModifier` wires those kernels to image buffers: it owns image dimensions, focal length, the per-shot interpolated coefficients, the unit-circle normalization, and `ApplyGeometryDistortion` / `ApplyColorModification` / `ApplySubpixelDistortion`.

Five tests are `#[ignore]`'d on this — they all assert `Modifier::apply_*` returns `true` (upstream's smoke tests don't pin numerical values, just exercise the pipeline). To unblock them, port:

- `libs/lensfun/modifier.cpp` — the constructor (`lfModifier::lfModifier`), `EnableDistortionCorrection`, `EnableTCACorrection`, `EnableVignettingCorrection`, and the per-pixel callback registration.
- The `lfCoordDistCallbackData` / `lfCoordCallbackData` / `lfColorCallbackData` plumbing — these hold the interpolated coefficients used by the kernels.
- `ApplyGeometryDistortion` / `ApplySubpixelDistortion` (in `modifier.cpp` too) — these iterate buffer rows and call the registered callbacks per pixel.
- Pixel ↔ normalized coordinate mapping (image dims + focal + crop → unit circle).

Estimated effort: 2-3 focused hours. Reference port is `mod-coord.cpp:1-200` for the coordinate setup and `modifier.cpp` for the buffer iteration.

## Known divergences from upstream

Each is intentional and documented at the call site. None are bugs.

1. **ACM models**: parsed by `db.rs` and stored as `None` for distortion/TCA/vignetting. Adding ACM means extending the `*Model` enums in `calib.rs` and the math kernels in `mod_coord` / `mod_subpix` / `mod_color`. Non-blocking; no Rust consumer has asked for ACM yet.
2. **Crop-factor loop in `MatchScore`** iterates over the lens's single `crop_factor` rather than upstream's `lfLensCalibrationSet` vector — Rust port has flat calibration Vecs by design.
3. **`find_cameras` / `find_lenses` signatures** are simplified relative to upstream (no `sflags`, no `maker` arg on `find_lenses`). Implementation matches upstream defaults.
4. **`fuzzy_str_cmp` casefolding** uses Rust `str::to_lowercase` instead of glib's case-fold tables. Functionally equivalent for ASCII, which is all the upstream test suite exercises.
5. **`Modifier` doesn't exist yet** beyond a stub. Tests that need it are `#[ignore]`'d with reasons.

## Risks for future sessions

1. **Modifier pixel/normalized mapping** is the trickiest part of the buffer integration. Upstream uses `NormScale` and `NormUnScale` (see `mod-coord.cpp` top-of-file). Get these wrong and every kernel call lands in the wrong place. Port test-driven from `test_modifier_regression.cpp` — that one pins specific pixel-output floats within `1e-3` tolerance.

2. **Float bit-exactness** continues to matter. Don't refactor accumulation order. The kernels in this session were ported faithfully; future ports should hold the same line.

3. **Multi-language strings**: parsed but not surfaced to query results yet. When `Camera::model_localized` etc. matter, wire them into `MatchScore` (it currently only matches against the default name).

4. **No end-to-end image test exists yet.** All current tests are scalar/buffer-level. Once `Modifier` lands, add at least one test that takes a small synthetic image, runs the full pipeline, and verifies a known pixel.
